# Creating Macros

Build automated workflows by chaining tool calls into reusable macros.

## What is a macro?

A macro chains multiple tool calls together into a repeatable workflow. Each step is a single tool call against a connected MCP server. Macros can run manually, on a schedule, or via webhook trigger.

## Creating a macro

1. Open the **Macros** tab from the bottom tab bar.
2. Tap **Create Macro**.
3. Enter a name and description.
4. Add steps by selecting tools from your connected servers.
5. Configure each step's parameters.
6. Optionally set up scheduling or webhook triggers.
7. Tap **Save** to create the macro.

> **Note:** Start with the smallest working version of your macro first. After the basic flow runs successfully, add conditions, scheduling, or webhook input.

## Macro steps

Each step in a macro is a single tool call. Steps run sequentially by default. Each step has these configurable fields:

| Field | Description |
| --- | --- |
| Tool | The server and tool to execute |
| Parameters | Static values or references to previous steps' outputs |
| Condition | Optional expression that determines whether the step runs |
| Timeout | Maximum execution time for this step (default: 30s) |

### Referencing previous step outputs

You can reference output from an earlier step using `{{steps[N].output}}` syntax. If step 1 creates a repository and returns its ID, step 2 can use `{{steps[0].output.id}}` to reference it.

Example macro definition:

```json
{
  "name": "Create issue and notify Slack",
  "description": "Create a GitHub issue and post a Slack notification about it",
  "steps": [
    {
      "tool": "github/create_issue",
      "parameters": {
        "owner": "acme-inc",
        "repo": "platform",
        "title": "Automated bug report",
        "body": "Generated by MCP Hub macro"
      }
    },
    {
      "tool": "slack/post_message",
      "parameters": {
        "channel": "#engineering",
        "text": "New issue created: {{steps[0].output.text}}"
      }
    }
  ]
}
```

<details>
<summary>Troubleshooting step output references</summary>

If a later step cannot resolve a reference, check these common causes:

1. The earlier step did not run because its condition evaluated to false.
2. The earlier step failed before returning output.
3. The reference path does not match the shape of the returned data.
4. The step index is zero-based, so the first step is `steps[0]`.

</details>

## Scheduling macros

Macros can run on a schedule using cron expressions. From the macro editor:

1. Tap **Schedule**.
2. Enter a cron expression.
3. Tap **Save**.

Common cron patterns:

| Pattern | Schedule |
| --- | --- |
| `0 * * * *` | Every hour |
| `0 9 * * 1-5` | Weekdays at 9 AM |
| `*/15 * * * *` | Every 15 minutes |
| `0 0 * * 0` | Weekly on Sunday at midnight |

> **Note:** Cron times use UTC. Adjust the expression to match your timezone.

## Webhook triggers

Macros can be triggered externally via webhook URL. From the macro editor:

1. Tap **Webhook**.
2. Enable the webhook trigger.
3. Copy the generated webhook URL.
4. Send a POST request to trigger the macro.

Example webhook payload:

```json
{
  "priority": "high"
}
```

The webhook body is available to macro steps as `{{webhook.body}}`.

> **Note:** Use webhook triggers when another system needs to start the macro automatically, such as an alerting tool or internal app.

## Running macros manually

To run a macro on demand:

1. Open the macro from the **Macros** tab.
2. Tap **Run**.
3. Watch the live execution progress.

Each step shows its status in real time: pending, running, completed, or error. If a step fails, the macro stops and shows the error.

## Viewing macro history

Past macro runs are visible in the **Analytics** tab. Each entry shows:

- Macro name
- Trigger type: manual, schedule, or webhook
- Start and end time
- Status: completed, failed, or partial
- Duration

<details>
<summary>What to check in a failed run</summary>

When a macro run fails, review the run history for:

1. The step where execution stopped.
2. The exact tool error returned by the connected server.
3. Any missing or malformed parameters.
4. Whether a referenced earlier output was empty or unavailable.
5. Whether the step hit its timeout limit.

</details>

## Managing macros

From the macro list view you can:

- **Edit** — Modify steps, parameters, schedule, or webhook settings
- **Duplicate** — Create a copy of an existing macro
- **Delete** — Remove a macro permanently
- **Enable/Disable** — Pause a macro without deleting it

> **Warning:** Deleting a macro removes its execution history. Export any history you need before deleting.
